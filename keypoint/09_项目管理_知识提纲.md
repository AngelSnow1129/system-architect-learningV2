## 🎯 核心知识点框架

### 重点1：进度管理与关键路径法 ⭐⭐⭐⭐⭐
**难度**：★★★★☆
**考试频率**：中（虽然近年考得少，但仍需掌握）

**进度管理过程：** ⭐⭐⭐⭐
```
1. 活动定义：确定完成项目各项可交付成果而需要开展的具体活动
2. 活动排序：识别和记录各项活动之间的先后关系和逻辑关系
3. 活动资源估算：估算完成各项活动所需要的资源类型和数量
4. 活动历时估算：估算完成各项活动所需要的具体时间
5. 进度计划编制：制订项目进度计划
6. 进度控制：根据进度计划开展项目活动，发现偏差则分析原因或调整
```

**进度图形描述方法：** ⭐⭐⭐⭐
```
1. Gantt图（甘特图）：
   - 横道图
   - 直观显示任务的开始和结束时间
   - 易于理解
   - 不能显示任务间的依赖关系

2. PERT图（项目计划评审技术图）：
   - 网络图
   - 显示任务间的依赖关系
   - 可以计算关键路径
   - 适合复杂项目
```

**关键路径法（CPM）详解：** ⭐⭐⭐⭐⭐

**核心概念：**
```
关键路径：
- 项目的最短工期
- 从开始到结束时间最长的路径
- 可能有多条关键路径
- 会随着活动变化而变化

关键活动：
- 关键路径上的活动
- 最早开始时间 = 最晚开始时间
- 总浮动时间 = 0
```

**四个关键时间：** ⭐⭐⭐⭐⭐

**1. 最早开始时间（ES - Earliest Start）：**
```
定义：某项活动能够开始的最早时间

计算方法（顺推）：
ES = 所有前置活动最早完成时间（EF）的最大值

起点活动：ES = 0
```

**2. 最早结束时间（EF - Earliest Finish）：**
```
定义：某项活动能够完成的最早时间

计算公式：
EF = ES + 活动持续时间
```

**3. 最迟结束时间（LF - Latest Finish）：**
```
定义：为了使项目按时完成，某项活动必须完成的最迟时间

计算方法（逆推）：
LF = 所有后续活动最晚开始时间（LS）的最小值

终点活动：LF = EF
```

**4. 最迟开始时间（LS - Latest Start）：**
```
定义：为了使项目按时完成，某项活动必须开始的最迟时间

计算公式：
LS = LF - 活动持续时间
```

**计算步骤：** ⭐⭐⭐⭐⭐

**步骤1：顺推计算ES和EF**
```
从起点开始，向终点推进
1. 起点活动：ES = 0
2. 其他活动：ES = max(所有前置活动的EF)
3. 所有活动：EF = ES + 持续时间
```

**步骤2：逆推计算LF和LS**
```
从终点开始，向起点推进
1. 终点活动：LF = EF
2. 其他活动：LF = min(所有后续活动的LS)
3. 所有活动：LS = LF - 持续时间
```

**步骤3：计算浮动时间**
```
总浮动时间 = LS - ES 或 LF - EF
自由浮动时间 = min(后续活动ES) - 本活动EF
```

**步骤4：确定关键路径**
```
总浮动时间 = 0 的活动是关键活动
关键活动连成的路径是关键路径
```

**浮动时间：** ⭐⭐⭐⭐⭐

**总浮动时间（Total Float）：**
```
定义：在不延误项目完工时间的前提下，活动可以推迟或拖延的时间量

计算公式：
总浮动时间 = LS - ES
或
总浮动时间 = LF - EF
或
总浮动时间 = 关键路径时长 - 本活动所在路径时长

特点：
- 关键活动的总浮动时间为0
- 非关键活动的总浮动时间 > 0
- 表示活动的进度灵活性
```

**自由浮动时间（Free Float）：**
```
定义：在不延误任何紧后活动的最早开始时间的前提下，
      活动可以推迟或拖延的时间量

计算公式：
自由浮动时间 = min(紧后活动的ES) - 本活动的EF

特点：
- 自由浮动时间 ≤ 总浮动时间
- 使用自由浮动时间不影响后续活动
```

**典型例题：** ⭐⭐⭐⭐⭐

**例题1：** 某项目网络图如下（单位：周），求关键路径和项目工期。

```
活动及持续时间：
A(2) → B(4) → E(4) → G(3)
A(2) → C(5) → E(4) → G(3)
A(2) → D(6) → F(6) → G(3)
```

**解答：**

**步骤1：列出所有路径及时长**
```
路径1：A-B-E-G = 2+4+4+3 = 13周
路径2：A-C-E-G = 2+5+4+3 = 14周
路径3：A-D-F-G = 2+6+6+3 = 17周
```

**步骤2：确定关键路径**
```
最长路径是路径3：A-D-F-G
关键路径：A-D-F-G
项目工期：17周
```

**答案：关键路径A-D-F-G，工期17周**

**例题2：** 基于例题1，求活动C的总浮动时间。

**解答：**
```
方法1：路径差值法
活动C所在路径：A-C-E-G = 14周
关键路径时长：17周
总浮动时间 = 17 - 14 = 3周

方法2：时间差值法（需要计算ES、EF、LS、LF）
活动C：
ES = 2（A完成后）
EF = 2 + 5 = 7
LS = 5（要保证E在9周开始）
LF = 5 + 5 = 10
总浮动时间 = LS - ES = 5 - 2 = 3周
```

**答案：3周**

**例题3：** 项目实施中，活动D延期2周，活动A实际工期6周，活动F提前1周，求新的项目工期。

**解答：**
```
原关键路径：A-D-F-G = 2+6+6+3 = 17周

调整后：
路径1：A-B-E-G = 6+4+4+3 = 17周
路径2：A-C-E-G = 6+5+4+3 = 18周
路径3：A-D-F-G = 6+(6+2)+(6-1)+3 = 6+8+5+3 = 22周

新的关键路径：A-D-F-G
新的项目工期：22周
```

**答案：22周**

**快速计算技巧：** ⭐⭐⭐⭐⭐

**技巧1：简单网络图直接列举法**
```
如果网络图不复杂，直接列出所有路径
计算每条路径的总时长
最长的路径就是关键路径
```

**技巧2：表格题转换法**
```
根据紧前活动关系画出网络图
然后列举所有路径
找出最长路径
```

**技巧3：总浮动时间快速计算**
```
总浮动时间 = 关键路径时长 - 活动所在最长路径时长
这个方法最快，不需要计算ES、LS等
```

### 重点2：软件配置管理 ⭐⭐⭐⭐
**难度**：★★★☆☆
**考试频率**：中

**配置管理定义：** ⭐⭐⭐⭐
```
定义：为了系统地控制配置变更，在系统的整个生命周期中
      维持配置的完整性和可跟踪性，而标识系统在不同时间点上
      配置的学科

目的：
- 控制配置变更
- 维持配置完整性
- 保证可跟踪性
```

**配置管理六大活动：** ⭐⭐⭐⭐⭐
```
1. 制订配置管理计划
2. 配置标识
3. 配置控制
4. 配置状态报告
5. 配置审计
6. 发布管理和交付
```

**配置项（CI）：** ⭐⭐⭐⭐⭐

**定义：**
```
为配置管理设计的硬件、软件或二者的集合，
在配置管理过程中作为一个单个实体来对待
```

**典型配置项：**
```
- 项目计划书
- 需求文档
- 设计文档
- 源代码
- 可执行代码
- 测试用例
- 运行软件所需的各种数据
```

**配置项主要属性：** ⭐⭐⭐⭐⭐
```
1. 名称
2. 标识符
3. 文件状态
4. 版本
5. 作者
6. 日期
```

**配置项分类：**
```
1. 基线配置项：
   - 所有的设计文档和源程序等
   - 向开发人员开放读取权限

2. 非基线配置项：
   - 项目的各类计划和报告等
   - 向PM、CCB及相关人员开放
```

**配置项状态：** ⭐⭐⭐⭐⭐

**三种状态：**
```
1. 草稿（Draft）：
   - 配置项刚建立时的状态
   - 可以随意修改

2. 正式（Formal）：
   - 配置项通过评审后的状态
   - 需要通过配置控制才能修改

3. 修改（Changing）：
   - 正式配置项被修改时的状态
   - 修改完毕并重新通过评审后变为"正式"
```

**状态转换：**
```
草稿 --评审通过--> 正式
正式 --开始修改--> 修改
修改 --评审通过--> 正式
```

**配置项版本号规则：** ⭐⭐⭐⭐⭐

**1. 草稿状态：0.YZ**
```
格式：0.YZ
YZ范围：01-99
说明：随着草稿修正，YZ递增

例：0.01 → 0.02 → 0.03 → ... → 0.99
```

**2. 正式状态：X.Y**
```
格式：X.Y
X范围：1-9（主版本号）
Y范围：0-9（次版本号）

规则：
- 第一次成为正式文件：1.0
- 小幅升级：增加Y值（1.0 → 1.1 → 1.2）
- 大幅升级：增加X值（1.9 → 2.0）
- Y增加到一定程度，X值增加

例：1.0 → 1.1 → 1.2 → ... → 1.9 → 2.0
```

**3. 修改状态：X.YZ**
```
格式：X.YZ
说明：正在修改时，只增大Z值，X.Y保持不变

规则：
- 修改过程中：增加Z值（1.01 → 1.02）
- 修改完毕成为正式：Z值设为0，增加X.Y值（1.02 → 1.1）

例：1.0 → 1.01 → 1.02 → 1.1
```

**版本号示例：** ⭐⭐⭐⭐⭐
```
开发过程：
0.01（草稿） → 0.02（草稿修改） → 1.0（第一次正式发布）
→ 1.01（修改中） → 1.1（小版本升级）
→ 1.11（修改中） → 1.2（小版本升级）
→ 2.0（大版本升级）
```

### 重点3：质量管理 ⭐⭐⭐⭐
**难度**：★★★☆☆
**考试频率**：低

**质量定义：** ⭐⭐⭐⭐
```
质量：软件产品特性的综合，表示软件产品满足明确（基本需求）
      或隐含（期望需求）要求的能力

质量管理：确定质量方针、目标和职责，并通过质量体系中的
          质量计划、质量控制、质量保证和质量改进来实现
```

**质量管理三大过程：** ⭐⭐⭐⭐⭐

**1. 质量规划：**
```
目的：识别项目及其产品的质量要求和标准

内容：
- 确定质量标准
- 制定质量计划
- 描述如何达到质量要求

时机：项目启动阶段
```

**2. 质量保证（QA）：**
```
目的：保证项目的质量

方法：
- 系统的质量审计（软件评审）
- 过程分析

特点：
- 定期进行（如每个阶段末）
- 关注过程
- 预防性质

活动：
- 过程审计
- 产品审计
- 质量体系审核
```

**3. 质量控制（QC）：**
```
目的：监控项目的具体结果，判断是否符合质量标准

方法：
- 实时监控
- 测试
- 检查
- 审查

特点：
- 持续进行
- 关注产品
- 检测性质

活动：
- 测试
- 缺陷跟踪
- 度量分析
- 制定纠正措施
```

**QA vs QC：** ⭐⭐⭐⭐⭐
```
质量保证（QA）：
- 关注过程
- 预防缺陷
- 定期进行
- 面向整个项目

质量控制（QC）：
- 关注产品
- 检测缺陷
- 持续进行
- 面向具体产品
```

### 重点4：风险管理 ⭐⭐⭐⭐
**难度**：★★★☆☆
**考试频率**：低

**风险管理过程：** ⭐⭐⭐⭐⭐

**1. 风险管理计划编制：**
```
制定如何安排与实施项目的风险管理
```

**2. 风险识别：**
```
目的：识别项目中已知和可预测的风险

内容：
- 确定风险来源
- 确定产生条件
- 描述风险特征
- 形成风险列表
```

**3. 风险定性分析：**
```
目的：对已识别的风险进行排序

内容：
- 确定风险可能性
- 确定风险影响
- 确定风险优先级
- 确定风险类型
```

**4. 风险定量分析：**
```
目的：量化风险发生的可能性和后果

方法：
- 灵敏度分析
- 期望货币价值分析
- 决策树分析
- 蒙特卡罗模拟
```

**5. 风险应对计划编制：**
```
目的：为每个风险制定应对措施

消极风险应对策略：
- 避免策略：改变计划，消除风险
- 转移策略：将风险转移给第三方
- 减轻策略：降低风险发生概率或影响

积极风险应对策略：
- 开拓策略：确保机会实现
- 分享策略：与第三方分享机会
- 提高策略：提高机会发生概率或影响
```

**6. 风险监控：**
```
目的：监控风险计划的执行

内容：
- 检测残余风险
- 识别新的风险
- 保证风险计划执行
- 评价计划有效性
```

**风险分类：** ⭐⭐⭐⭐⭐

**1. 项目风险：**
```
内容：
- 预算问题
- 进度问题
- 人员问题
- 资源问题
- 用户和需求问题
- 项目复杂性、规模和结构的不确定性

影响：
- 拖延项目进度
- 增加项目成本
```

**2. 技术风险：**
```
内容：
- 设计问题
- 实现问题
- 接口问题
- 测试和维护问题
- 规格说明的多义性
- 技术上的不确定性
- 技术陈旧或不成熟

影响：
- 威胁系统质量
- 威胁交付时间
- 开发困难或不可能
```

**3. 商业风险：**
```
五种商业风险：

1. 市场风险：
   - 系统虽优秀但不是市场真正想要的

2. 策略风险：
   - 系统不再符合企业信息系统战略

3. 销售风险：
   - 销售部门不清楚如何推销系统

4. 管理风险：
   - 失去上级管理部门的支持

5. 预算风险：
   - 没有得到预算或人员保证
```

**快速记忆法：** ⭐⭐⭐⭐⭐

**关键路径：**
```
"最长路径是关键，关键活动浮动零"
"顺推ES和EF，逆推LF和LS"
"总浮动看路径差，自由浮动看后续"
```

**配置项状态：**
```
"草正修，评审转"
草稿 → 正式 → 修改 → 正式
```

**配置项版本号：**
```
"草稿0.YZ，正式X.Y，修改X.YZ"
"小改Y增，大改X增"
```

**质量管理：**
```
"规划定标准，保证重过程，控制看产品"
```

**风险类型：**
```
"项技商"
项目风险、技术风险、商业风险
```

**易错点提醒：** ⭐⭐⭐⭐⭐
1. 关键路径是最长路径，不是最短路径
2. 关键活动的总浮动时间为0
3. 总浮动时间 = LS - ES 或 LF - EF
4. 配置项版本号：草稿0.YZ，正式X.Y，修改X.YZ
5. 质量保证关注过程，质量控制关注产品
6. 风险应对：消极风险用避免、转移、减轻
7. 自由浮动时间 ≤ 总浮动时间
